Rust 引入 `::<>`（涡轮鱼语法）是为了彻底解决以下三个痛点：
## 1. 彻底消除语法歧义（避免 C++ 的深坑）
在 C++ 中，由于没有 `::` 分隔，编译器在遇到 < 时，必须先查阅上下文符号表才能确定它是“小于号”还是“模板参数”。
这会导致极其荒谬的解析歧义。例如以下伪代码：

```cpp
// 在 C++ 中，如果编译器不看到前面的声明，它根本不知道这行代码在干嘛：
result = level.parse < u32 > (x);
```

* 可能 1：调用模板函数 parse，参数是 x。
* 可能 2：`level.parse` 是一个数字，表达式意思是 (`level.parse < u32) > (x`)——即两个连续的数学比较！

为了分辨这两种情况，C++ 编译器不得不引入极其复杂的双阶段查找（Two-phase lookup）和 typename、template 关键字来给编译器“当保姆”。而 Rust 通过加一个 `::`，让编译器仅靠这一行代码的语法结构（无需上下文）就能 100% 确定这是泛型。
## 2. 保证单向从左到右解析（提升编译速度）
Rust 的设计目标之一是让词法分析器可以不依赖上下文、单向地（One-pass）解析代码。

* C++ 遇到 a < b > 时，必须暂停，去前面找 a 和 b 的定义。
* Rust 遇到 `.parse::<u32>()` 时，看到 :: 就能闭着眼直接往前走，极大地简化了编译器的设计，也避免了复杂的隐式类型二义性。

## 3. C++ 的妥协：`template` 关键字
事实上，C++ 在某些特定场景下，也必须加类似的标记，只是它写成了关键字。

在 C++ 的类模板内部调用另一个类的成员模板函数时，你也必须写出丑陋的 `.` 后面跟 `template` 的语法：

```cpp
// C++ 的动态模板调用，必须加 template 关键字，否则编译报错！
obj.template parse<u32>(); 
```

Rust 的 ::<> 统一了这种场景，无论是普通泛型函数还是关联函数，规矩全部一视同仁。
