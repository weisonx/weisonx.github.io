# C++ STL并发接口

高级接口：

`async()` 启动线程

`class future<>` 处理返回值和异常

低层接口：

`class thread` 启动线程
`class promise` 处理返回值和异常

同步机制：

	锁	`class mutex`等

	原子操作	`class atomic<>`

	条件变量	`class condition_variable`

# 多线程框架与Qt对比

在线程框架的设计上，C++ STL 走的是“工具箱”路线（提供零件），而 Qt 走的是“生产线”路线（提供流程）。
以下是两者在线程生命周期管理、任务调度和通信机制上的异同：
## 1. 线程生命周期的管理（核心异同）

* C++ STL (std::thread)：显式管理
* 行为：创建一个 std::thread 对象即启动一个系统线程。
   * 约束：你必须在对象销毁前显式调用 .join()（等待完成）或 .detach()（后台运行）。如果忘记调用，程序会直接 std::terminate 崩溃。
   * 状态：STL 线程没有内置的“状态查询”机制（比如是否正在运行），也没有内置的停止请求（直到 C++20 引入 std::jthread 才有 stop_token）。
* Qt (QThread)：对象模型管理
* 行为：QThread 是一个管理线程的控制类。线程不一定在构造时启动，需手动调用 .start()。
   * 自动清理：可以通过信号槽绑定 finished 信号到 deleteLater 槽，实现线程运行完自动销毁。
   * 事件循环：每个 QThread 默认启动一个 QEventLoop（事件循环），这使得线程可以接收定时器事件、网络回调等。

## 2. 任务派发与执行模式

* STL：函数式驱动
* 通常通过传递 Lambda 表达式或函数指针来执行任务。
   * std::async 提供了简单的异步任务封装，但缺乏复杂的线程池调度策略。
* Qt：两种主流模式
  * 继承模式：重写 QThread::run()，适合传统的、不依赖事件循环的长耗时计算。
  * Worker 模式（推荐）：创建一个继承自 QObject 的 Worker 类，使用 moveToThread() 将其“推”入某个线程。这种模式下，Worker 的函数可以通过信号触发，执行过程依然保留在目标线程中。

## 3. 线程间通信（最大的差异点）

* STL：共享内存 + 同步锁
* 主要靠 std::mutex、std::condition_variable 或原子操作。
   * 通信是侵入式的：你需要手动保护数据，设计并发安全队列。
* Qt：信号槽（Signals & Slots）
   * 异步解耦：通过 Qt::QueuedConnection 发送信号，参数会被自动拷贝并压入目标线程的事件队列。
   * 类型安全：不需要手动写锁，开发者只需要关注信号和槽的连接，底层同步由 Qt 框架处理。

## 4. 线程池与高层抽象

* STL：标准库没有官方线程池（直到目前的 C++23 仍未正式标准化标准线程池），开发者通常需要手写或使用第三方库。
* Qt：内置 QThreadPool 和 QtConcurrent。
* QtConcurrent::run 会自动从全局线程池取线程，处理完后回收，非常适合大规模并行计算。

## 总结对照

| 特性 | C++ STL 线程框架 | Qt 线程框架 |
|---|---|---|
| 层级 | 底层封装，无额外开销 | 高层抽象，集成度高 |
| 基础单位 | std::thread / std::jthread | QThread / QRunnable |
| 通信手段 | 共享变量、Promise/Future | 信号槽（跨线程通信） |
| 事件驱动 | 无（需自行实现） | 有（内置事件循环） |
| 适用场景 | 追求极致性能、底层库、跨框架开发 | 桌面应用、异步IO多、逻辑复杂的业务 |
