在 C++ Qt 程序中，要实现程序崩溃时自动生成 Dump 文件，最标准且跨平台（主要针对 Windows 和 Linux）的方法是集成谷歌开源的 Crashpad（Breakpad 的升级版）或 Breakpad。

------------------------------
## 方案一：集成 Google Breakpad（主流、跨平台）
Breakpad 会接管系统的崩溃信号（如 Windows 的 SEH 或 Linux 的 SIGSEGV），并在崩溃瞬间将内存状态写入 .dmp 文件。
## 1. 源码引入与编译
首先从 GitHub 官方仓库 下载 Breakpad 源码，并将其作为子模块或静态库编译进你的 Qt 项目。
## 2. 在 Qt main.cpp 中初始化
在 QApplication 启动前，初始化 Breakpad 的崩溃处理器（以 Windows 环境为例）：

```
#include <QApplication>#include <QDir>#include "client/windows/handler/exception_handler.h" // 引入 Breakpad 头文件
// 崩溃回调函数bool dumpCallback(const wchar_t* dump_path, const wchar_t* minidump_id,
                  void* context, EXCEPTION_POINTERS* exinfo,
                  MDRawAssertionInfo* assertion, bool succeeded) 
{
    // 崩溃后执行的代码（注意：此时堆内存可能已损坏，避免执行复杂 Qt UI 操作）
    qDebug() << "程序崩溃！Dump 文件已保存至路径:" << QString::fromWCharArray(dump_path);
    return succeeded;
}
int main(int argc, char *argv[]) 
{
    QApplication a(argc, argv);

    // 1. 创建用于存放 Dump 文件的文件夹
    QString dumpPath = QApplication::applicationDirPath() + "/crashes";
    QDir().mkpath(dumpPath);

    // 2. 实例化 Breakpad 处理器
    google_breakpad::ExceptionHandler eh(
        dumpPath.toStdWString(),
        nullptr,
        dumpCallback,
        nullptr,
        google_breakpad::ExceptionHandler::HANDLER_ALL
    );

    // 你的其他 Qt 业务逻辑
    // ...

    return a.exec();
}
```

## 3. 构建配置 (CMake / qmake)
为了让 Dump 文件能够还原出堆栈代码行号，编译时必须生成调试符号。

* CMake 配置 (CMakeLists.txt)：

```
# 确保 Release 模式下也生成调试符号 (PDB 文件)
set(CMAKE_CXX_FLAGS_RELEASE "${CMAKE_CXX_FLAGS_RELEASE} /Zi")
set(CMAKE_EXE_LINKER_FLAGS_RELEASE "${CMAKE_EXE_LINKER_FLAGS_RELEASE} /DEBUG")
```

* qmake 配置 (.pro)：

CONFIG += force_debug_info
CONFIG += separate_debug_info

## 方案二：使用 Windows 原生 API（仅限 Windows，轻量级）

如果你的程序只在 Windows 上运行，不需要引入第三方库，直接调用系统的 SetUnhandledExceptionFilter 即可捕获崩溃并写入 MiniDump。

## 1. 编写原生捕获代码
新建一个 crash_handler.h，利用微软的 DbgHelp.dll 来写文件：

```
#pragma once#include <windows.h>#include <dbghelp.h>#include <string>
// 显式链接 DbgHelp 库#pragma comment(lib, "DbgHelp.lib")
LONG WINAPI WinCrashHandler(EXCEPTION_POINTERS* exceptionInfo) 
{
    // 创建或者打开 dump 文件
    HANDLE hFile = CreateFileA("crash.dmp", GENERIC_WRITE, 0, NULL, 
                               CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, NULL);
    
    if (hFile != INVALID_HANDLE_VALUE) {
        MINIDUMP_EXCEPTION_INFORMATION dumpInfo;
        dumpInfo.ThreadId = GetCurrentThreadId();
        dumpInfo.ExceptionPointers = exceptionInfo;
        dumpInfo.ClientPointers = TRUE;

        // 写入 MiniDump
        MiniDumpWriteDump(GetCurrentProcess(), GetCurrentProcessId(), hFile, 
                          MiniDumpNormal, &dumpInfo, NULL, NULL);
        CloseHandle(hFile);
    }
    
    // 弹窗提示用户，或者返回 EXCEPTION_EXECUTE_HANDLER 退出程序
    MessageBoxA(NULL, "程序发生严重错误已崩溃，Dump 文件已生成。", "崩溃提示", MB_OK | MB_ICONERROR);
    return EXCEPTION_EXECUTE_HANDLER;
}
```

## 2. 在 Qt main.cpp 中注册

```
#include <QApplication>#include "crash_handler.h"
int main(int argc, char *argv[]) 
{
    // 注册 Windows 全局异常捕获过滤器
    SetUnhandledExceptionFilter(WinCrashHandler);

    QApplication a(argc, argv);
    // ... 
    return a.exec();
}
```

------------------------------
## ⚠️ Qt 环境下的核心注意事项

   1. 不要在崩溃回调中调用复杂的 Qt UI 逻辑：当 dumpCallback 或 WinCrashHandler 被触发时，程序已经处于不稳定状态（堆栈可能已断开）。此时严禁调用 QMessageBox（方案二中的 Windows 原生 MessageBoxA 是安全的）、发送 Qt Signal 或操作 UI 控件，否则极易引发二次崩溃，导致 Dump 文件写入失败。
   2. 多线程崩溃捕获：以上两种方案默认都能捕获子线程（QThread）中未处理的严重致命异常（如野指针访问、内存越界）。但如果子线程的异常被线程内部吞掉了，则不会触发全局崩溃。
   3. 发布产品时的文件配套：程序发布时，.exe 丢给用户；而编译生成的 .pdb（Windows 调试符号文件） 必须妥善保存在开发者的电脑上。用户端程序崩溃生成 .dmp 后，你必须同时拥有对应版本的 .exe + .dmp + .pdb，才能在 Visual Studio 中双击打开 .dmp 并完美定位到崩溃的那一行 C++ 代码。
