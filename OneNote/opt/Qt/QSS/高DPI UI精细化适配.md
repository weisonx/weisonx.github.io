## 1. 方案对比：从“手动挡”到“半自动挡”

* 静态多 QSS 方案：
* 操作：针对不同字号预设多套 QSS 文件（手动调校 Padding ±2px）。
   * 核心价值：解决了“线性缩放”不符合审美的问题，防止 UI 在高 DPI 下显得“虚胖”。
   * 痛点：维护成本高，改一个颜色要动所有文件。
* 动态变量 QSS 方案：
* 操作：只维护一套 QSS 模板，将关键数值设为占位符（如 @padding）。
   * 核心价值：逻辑与样式分离。通过代码动态计算数值并执行字符串替换。
   * 进阶点：实现了非线性缩放（文字可以大，但间距要克制增加）。

## 2. 面试话术：如何包装“手动适配”经历？
当面试官问到高 DPI 或 UI 适配时，可以这样表达你的思考深度：

“在处理高 DPI 适配时，我发现开启系统原生自动缩放（AA_EnableHighDpiScaling）虽然能解决清晰度，但会导致 UI 比例失调，界面变得松散。
为了追求极致的视觉控制，我采用了手动适配方案：

   1. 反思暴力缩放：主动关闭自动缩放，由程序接管像素控制。
   2. 精细化控制：文字放大时，间距不按比例翻倍，而是通过 ±2px 级的手动微调，确保高分辨率下界面依然紧凑。
   3. 样式分级：建立了基于字号的样式管理逻辑，保证了比例的绝对美观。”

## 3. 架构优化建议：字符串模板化
为了减少文件维护量，可以将你的“多文件逻辑”合并为“单模板逻辑”：

// 1. 定义 QSS 模板（使用占位符）
QString qssTemplate = "QLabel { font-size: %1px; padding: %2px; }";
// 2. 编写适配逻辑int fontSize = userSelectedSize; int padding = (fontSize > 16) ? 6 : 8; // 字号越大，间距增长越克制
// 3. 动态渲染应用
qApp->setStyleSheet(qssTemplate.arg(fontSize).arg(padding));

## 4. 核心感悟

* 不盲从原生 API：感知到“线性缩放不符合审美”是迈向高级 UI 开发者的重要标志。
* 像素级控制：在追求精致感的项目中，手动介入（或代码干预变量）的价值远高于系统自动生成的 UI。
